# Project Deployer

帮助用户将GitHub项目、博客教程、视频教程等内容部署到本地或远程服务器。

## 触发条件

当用户提到以下内容时，必须使用此skill：
- 部署、安装、运行项目
- GitHub链接、博客链接、视频链接
- 项目名称 + 部署/安装/运行
- 任何想要在服务器上部署项目的需求

## 完整执行流程

### 步骤1：获取项目内容

根据输入类型获取相关内容：

**GitHub链接**
- 使用WebFetch获取仓库信息
- 读取README.md获取部署说明
- 检查项目根目录的配置文件（package.json, Cargo.toml, requirements.txt, Dockerfile, docker-compose.yml, Makefile等）

**博客/教程链接**
- 使用WebFetch获取页面内容
- 提取关键的安装/部署命令

**视频链接**
- 提取视频标题和描述
- 使用WebSearch搜索相关的文字教程
- 将搜索结果展示给用户参考

**项目名称**
- 搜索GitHub获取相关项目
- 让用户确认具体是哪个项目

### 步骤2：提取部署方案

从获取的内容中识别所有可能的部署方案：

1. 列出所有找到的部署方式（如：直接运行、Docker部署、PM2部署、云平台部署等）
2. 每个方案附上简要说明
3. 标注每个方案需要的依赖和环境

### 步骤3：用户选择方案

将所有方案以清晰的方式展示给用户：
```
可选部署方案：
1. [方案名称] - 适用场景：xxx，需要：xxx
2. [方案名称] - 适用场景：xxx，需要：xxx
...
```

等待用户选择其中一个方案。

### 步骤4：环境检测

检测当前运行环境：
- 操作系统（Linux/macOS/Windows）
- 已安装的运行时（Node.js/Python/Go/Java/Docker等）
- 端口占用情况
- 必要工具是否已安装

### 步骤5：确认执行模式

**模式A：命令预览（默认）**
- 输出完整的安装和运行命令
- 用户手动复制执行
- 遇到问题可以随时问AI

**模式B：自动执行**
- 用户必须说"执行"才进入自动执行模式
- 不说"执行"则保持命令预览模式

### 步骤6：执行部署

#### 命令预览模式
输出格式：
```
=== [方案名称] ===

# 安装步骤
1. [命令1]
2. [命令2]
...

# 运行步骤
1. [命令1]
2. [命令2]
...

# 注意事项
- [提醒1]
- [提醒2]
```

#### 自动执行模式

执行流程：
1. **执行前确认**：再次让用户确认是否开始执行
2. **逐步执行**：每个命令执行后简要汇报结果
3. **出错处理**：
   - 自动尝试回滚（最多3次）
   - 回滚3次仍失败，暂停并等待用户指示
   - 每次回滚前汇报当前状态
4. **执行完成**：汇报最终状态和服务运行情况

执行输出格式：
```
> [命令]
✓ 成功 / ✗ 失败
[简要结果]
```

## 远程服务器支持

用户可以选择部署到远程服务器：

### 需要用户提供的配置
- 服务器地址（IP/域名）
- SSH端口（默认22）
- 用户名
- 认证方式（密码/SSH密钥）

### 远程执行注意事项
- 不保存凭据，每次让用户输入
- 执行前先测试SSH连接
- 远程执行时每个命令都简要汇报

## 边界情况处理

### 复杂项目
遇到以下情况时，提示用户：
> "此项目较为复杂（涉及多服务/数据库编排/特殊配置），我会尽力协助你。如果遇到需要手动操作的步骤，我会标注提醒你。"

### 手动步骤
在命令中标注 `[需手动]`：
```
# 安装步骤
1. 配置数据库连接 [需手动]
2. npm install
3. npm run migrate [需手动]
```

### 依赖缺失
检测到缺失依赖时：
1. 先询问用户是否安装
2. 给出安装命令
3. 继续执行前确认

### 端口冲突
检测到端口被占用时：
1. 列出占用端口的进程
2. 询问用户是否停止或换端口

## 用户交互规范

### 必须遵守的规则

1. **不自动执行**：必须用户明确说"执行"才进入自动执行模式
2. **命令预览优先**：默认只输出命令，等用户选择
3. **风险命令警告**：删除/格式化/修改系统配置等命令前必须提醒
4. **出错暂停**：自动执行失败后必须等待用户指示
5. **简洁汇报**：自动执行时只汇报关键结果，不输出完整日志

### 沟通风格

- 保持简洁，不说废话
- 技术术语适当解释
- 遇到问题给出选项让用户选
